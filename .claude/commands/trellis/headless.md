# Headless Mode - 无头自动执行模式

进入无头模式（Headless Mode），自动执行任务直到完成，无需人工确认。

## 模式特性

### 1. 自动工具调用（YOLO 模式）
- 非敏感命令自动执行，无需确认
- 敏感命令（git push、删除文件等）仍需确认
- 提高自动化效率

### 2. 自动修复循环
- 自动执行 `修改 -> 编译 -> 运行 -> 验证 -> 修复` 循环
- 遇到错误自动分析并修复
- 无需人工介入

### 3. 熔断机制
- 针对同一错误的修复尝试不超过 **3次**
- 超过限制立即停止，输出"卡点分析报告"
- 避免无限循环

### 4. 会话管理
- 自动保存所有操作历史
- 支持断点恢复
- 输出 SESSION_ID 用于后续追踪

## 执行流程

### Phase 1: 任务分析
1. 理解用户需求
2. 确定执行步骤
3. 识别潜在风险点

### Phase 2: 自动执行
1. 按步骤执行任务
2. 遇到错误自动修复（最多3次）
3. 记录所有操作日志

### Phase 3: 验证完成
检查 DoD（Definition of Done）：
- [ ] 代码已保存且编译通过
- [ ] 已实际运行（无编译报错）
- [ ] 日志已验证（关键逻辑输出正确）
- [ ] 功能符合预期
- [ ] 无运行时错误
- [ ] 现场已清理（移除调试代码）

### Phase 4: 输出报告
```markdown
## 执行报告

### 任务描述
[任务内容]

### 执行步骤
1. [步骤1] - ✓ 完成
2. [步骤2] - ✓ 完成
3. [步骤3] - ⚠ 需要人工确认

### 修复记录
- 错误1: [描述] -> 修复方案: [方案] -> 结果: ✓
- 错误2: [描述] -> 修复方案: [方案] -> 结果: ✓

### DoD 检查
- [x] 编译通过
- [x] 运行验证
- [x] 日志验证
- [x] 功能正确
- [x] 无运行时错误
- [x] 现场清理

### 会话信息
SESSION_ID: [uuid]
```

## 熔断触发条件

当满足以下任一条件时，立即停止并输出卡点分析：

1. **重复错误**: 同一错误修复失败 3 次
2. **循环依赖**: 检测到修复逻辑陷入循环
3. **超时**: 单个步骤执行超过 10 分钟
4. **敏感操作**: 需要人工确认的操作

## 卡点分析报告格式

```markdown
## 🛑 熔断触发 - 卡点分析报告

### 触发原因
[重复错误/循环依赖/超时/敏感操作]

### 当前状态
- 执行步骤: [步骤名称]
- 尝试次数: [次数]
- 最后错误: [错误信息]

### 已尝试方案
1. [方案1] - 失败原因: [原因]
2. [方案2] - 失败原因: [原因]
3. [方案3] - 失败原因: [原因]

### 需要人工介入
- [ ] 检查配置: [具体配置项]
- [ ] 确认权限: [权限问题]
- [ ] 提供信息: [需要的信息]

### 恢复方式
使用 SESSION_ID=[uuid] 恢复会话后继续
```

## 使用场景

### ✓ 适合使用无头模式
- 批量文件处理
- 代码重构（明确范围）
- 自动化测试修复
- 依赖更新
- 文档生成

### ✗ 不适合使用无头模式
- 架构设计决策
- 需要用户选择的场景
- 涉及敏感数据操作
- 不确定需求的探索性任务

## 注意事项

1. **验证优先**: 所有修改必须通过运行验证，不能只看编译通过
2. **日志必查**: 必须查看实际运行日志，确认功能正确
3. **禁止占位**: 禁止 `return true` 等占位实现
4. **最小修改**: 只做必要修改，避免过度工程
5. **记录经验**: 任务完成后更新 LESSONS_LEARNED.md

## 退出无头模式

当需要人工决策时，自动退出无头模式并：
1. 输出当前状态
2. 说明需要决策的问题
3. 提供可选方案
4. 等待用户指令
