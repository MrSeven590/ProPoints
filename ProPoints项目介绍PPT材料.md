# 白酒制曲车间工分管理系统（ProPoints）项目介绍

> **项目定位**：基于 uni-app x 的离线优先移动端工分管理系统，面向白酒制曲五大作业环节设计，当前支持安曲、一翻、二翻、拆曲等核心场景下的工分录入、自动计算、跨仓扣减、考核扣分、公示截图、统计查询，以及 JSON/CSV 形式的数据导出。

---

## 一、案例真实性分析

### 1.1 业务痛点来源真实

**问题背景**：

- 白酒制曲车间采用传统人工记录工分方式，存在**记录不准确、统计困难、易出错**等问题
- 五大制曲作业环节（安曲/一翻/二翻/拆曲/堆曲）涉及**多仓、多人员、复杂计算**，人工管理效率低下
- 跨仓作业、轮次管理、技能系数等**业务规则复杂**，难以标准化执行

**真实场景验证**：

- 系统设计基于**白酒制曲车间实际工作流程**
- 解决**曲坯数复用、跨仓工分抽取、晾堂池子计算**等真实业务难题
- 支持**现场手套操作**（大触控区域、高对比度设计），符合车间实际使用环境

### 1.2 问题导向明确

| 业务痛点         | 解决方案                 | 实际价值       |
| ---------------- | ------------------------ | -------------- |
| 人工记录易出错   | 自动计算 + 实时校验      | 数据准确率提升 |
| 跨仓工分计算复杂 | 最大余数法自动分配       | 计算效率提升   |
| 统计查询困难     | 多维度查询 + JSON/CSV 导出 | 管理效率提升   |
| 公示流程繁琐     | 一键公示 + 截图分享      | 协同效率提升   |

## 二、提质增效成效分析

### 2.1 效率提升指标

**工分录入效率**：

- **自动化计算**：BIN 总分 = `floor1(kojiCount/20 × coef) × 10`，SESSION 池子 = `floor1(kojiCount/160 × coef)`
- **曲坯数复用**：一翻/二翻/拆曲自动从安曲记录带出，减少重复录入
- **智能推断**：仓号自动推断、轮次自动判断、人员默认沿用

**跨仓处理效率**：

- **最大余数法**：跨仓工分按曲坯数比例自动分配，保证总和恒等
- **实时预览**：修改工分后来源分摊实时更新，便于现场核对

**统计查询效率**：

- **多维度查询**：按人员、按仓、按日期、按月统计
- **索引加速**：`pp:idx:person:*` 和 `pp:idx:bin:*` 避免全量扫描
- **数据导出**：当前支持 JSON 备份导出/导入与 CSV 明细分享导出，便于归档与跨设备迁移

### 2.2 流程优化点

**数据验证流程**：

- **15 条校验规则**：structure → permission → business → consistency 四阶段执行
- **实时对账**：录入页底部实时展示对账状态和错误列表
- **提交拦截**：不平衡/缺字段时提交按钮禁用

**公示流程优化**：

- **一键公示**：触发公示后记录时间戳，不产生锁定效果
- **截图友好**：公示页布局适合截图发送群，无需横向滚动
- **口径切换**：支持原始/扣分/最终三种工分口径展示

### 2.3 协同改进

**人员协同**：

- **花名册管理**：当前支持粘贴导入与拼音索引搜索，OCR 导入仍为后续扩展方向
- **技能系数**：版本化管理，支持建议分配和自动平衡
- **审计字段**：记录创建者/最后编辑者，便于追溯

**数据协同**：

- **离线优先**：一期以 `uni.*Storage` 作为主存储，支持离线使用
- **本地 JSON**：会话数据结构化存储，便于二期迁移
- **备份恢复**：JSON 导出/导入，支持跨设备迁移

## 三、安全合规性分析

### 3.1 数据安全

**存储安全**：

- **本地存储**：一期数据完全存储在本地，不上传云端
- **Key 规范**：`pp:session:{date}:{stage}:{roundId}` 结构化存储，避免数据混乱
- **版本控制**：JSON 备份包含版本号和时间戳，便于回滚

**操作安全**：

- **二次确认**：关键操作（提交/公示/导出）有二次确认或明确反馈
- **审计追溯**：记录 `created_by_manager`、`updated_by_manager`、`updated_at` 等字段
- **历史可改**：已提交/已公示数据仍允许编辑，但保留修改记录

### 3.2 平台合规

**小程序限制适配**：

- **单 key 限制**：按“日期/环节/轮次”拆 key，避免单 key 过大
- **总量控制**：索引维护避免全量扫描，控制存储体积
- **性能优化**：大量录入场景下优化查询与存储

**用户体验合规**：

- **手套操作**：大触控区域、高对比度设计，符合车间环境
- **输入容错**：自动格式化为 1 位小数，不合法输入可被纠正
- **提示明确**：关键操作有明确提示和操作反馈

### 3.3 业务合规

**工分规则合规**：

- **精度统一**：统一 0.1 分精度，内部使用整数单位避免浮点误差
- **规则配置化**：环节、岗位、系数等均可配置，适应不同车间需求
- **轮次管理**：隔轮轮换微机计分权，保证公平性

**考核合规**：

- **扣分独立**：扣分记录独立存储，最终分不回写原始分
- **原因必填**：扣分 > 0 时原因必填，保证可追溯
- **三段口径**：原始/扣分/最终三种口径，满足不同统计需求

## 五、表达呈现效果

### 5.1 项目结构逻辑

**目录结构清晰**：

```text
ProPoints/
├── pages/          # 页面层：工作台、统计、我的
├── components/     # 组件层：业务组件、通用组件
├── domain/         # 领域层：models、services、stores
├── storage/        # 存储层：key规范、repository、初始化
├── utils/          # 工具层：通用工具函数
└── static/         # 静态资源
```

**技术栈明确**：

- **前端框架**：uni-app x（Vue 3 + UTS / uvue）
- **存储方案**：当前以 `uni.*Storage` 为主，本地离线运行；SQLite 可作为后续演进方向评估
- **构建工具**：HBuilderX + 自定义编译脚本

### 5.2 核心业务逻辑展示

**工分计算逻辑**：

```text
// BIN 总分计算
floor1(kojiCount / 20 × coef) × 10

// SESSION 池子计算（安曲晾堂等会话场景）
floor1(kojiCount / 160 × coef)

// 跨仓比例抽取（最大余数法）
按曲坯数比例分配，保证总和恒等
```

**数据验证逻辑**：

```text
// 四阶段执行
structure → permission → business → consistency

// 关键校验点
BIN 平衡：bin.total = sum(BIN 原始分) + sum(被跨仓抽取)
CROSS_BIN 来源完整：assignment.points_units = sum(source_points_units)
SESSION 池子平衡：pool_total = sum(SESSION 原始分)
```

### 5.3 技术亮点展示

**整数单位系统**：

- 内部统一使用 `points_units = points × 10`
- 避免浮点误差，保证计算精度
- UI 层与数据层分离，便于维护

**离线优先架构**：

- 一期完全离线，支持车间无网络环境
- 二期预留云端同步字段，渐进式升级
- 本地 JSON 结构化存储，便于迁移

**智能推断机制**：

- 仓号自动推断（历史最大 seq + 楼层默认）
- 轮次自动判断（隔轮轮换微机计分权）
- 人员默认沿用（晾堂/堆曲场景）
